# i18n Readiness — Flizow

**Status:** v1 ships English-only by deliberate choice. This doc is the answer to the future question "can we add Spanish / Japanese / Arabic?" — written so the cost is concrete instead of hand-wavy.

**Audited:** 2026-04-26 (Wave 10).

---

## TL;DR

Full i18n (string extraction + locale-aware formatting + RTL) is **3-4 weeks of focused work**, dominated by string extraction (~800-1000 strings across 50+ components) and the RTL CSS rewrite. None of it requires architectural changes — but none of it is small either.

If the question is "can we add Spanish *for marketing pages only*?" the answer is much smaller — those don't exist yet, and when they do, they're separate from the app.

---

## What's already i18n-friendly

- **Date formatting** — every `toLocaleDateString` / `toLocaleTimeString` call passes `undefined` for the locale arg, so the user's browser locale wins out of the box. The 8 date helpers in `src/utils/dateFormat.ts` follow the same pattern.
- **Number thousands separators** — `n.toLocaleString()` with no locale arg respects browser locale.
- **Direction-neutral icons** — most decorative SVGs are symmetric; the few directional ones (chevrons) flip via `transform: rotate()` rather than positional changes.

## What blocks i18n

### HIGH-cost blockers

| Blocker | Effort | Why |
|---|---|---|
| **String extraction (~800-1000 strings)** | 2-3 weeks | Strings are scattered across 50+ component files as JSX text nodes, `aria-label` attributes, placeholders, error messages, and empty-state copy. Top 5 files (FlizowShareModal, StatsTab, NotesTab, TouchpointsTab, FlizowAccountModal) hold ~35%. Needs a string-extraction tool (`babel-plugin-i18next` or similar) + a translation management workflow (Crowdin / Lokalise / Phrase). |
| **RTL (Arabic, Hebrew)** | 4-8 hours | 20+ directional CSS rules use `left:` / `right:` / `padding-left:` / `margin-right:` / `border-left:`. None use logical properties (`inset-inline-start`, `padding-inline-start`, `border-inline-start`). Tailwind utility classes in TSX (`ml-5`, `pr-4`) are also directional. RTL needs a sweep through every `*-left` and `*-right` rule. |
| **Pluralization** | 1 week | Plurals today are hardcoded ternaries: `count === 1 ? 'day' : 'days'`. English has 2 forms; Arabic has 3, Russian / Polish 3, Czech 4. Needs a CLDR plural-rules library (`Intl.PluralRules`, or i18next's plural backend). |

### MED-cost blockers

| Blocker | Effort | Why |
|---|---|---|
| **Currency** | 2 days | `formatMrr()` in `dateFormat.ts` hardcodes `'en-US'` + `'USD'`. `StatsTab` prefixes raw `'$'`. Multi-currency means: add currency field to user / client preferences, replace hardcodes, switch all currency formatters to `Intl.NumberFormat`. |
| **Relative-time prose** | 3 days | "Today", "Yesterday", "Tomorrow", "In 1 day", "Just now", etc. are hardcoded English in `NotesTab.tsx`, `TouchpointsTab.tsx`, `OverviewPage.tsx`. Each needs to either use `Intl.RelativeTimeFormat` or be added to the translation bundle. |
| **Week-start preference** | n/a | Removed 2026-04-26. The Sun/Mon segment in Account Settings was a meaningless setting because the Schedule grid is Mon–Fri-only by design (weekends skipped). If a Sat/Sun-aware view ships later, the preference should travel with that view, not live as a global setting. |

### LOW (already shipped or trivial)

| Blocker | Status |
|---|---|
| **CSS-baked string** ("No cards yet — drag one in from another list.") | ✅ Shipped 2026-04-26 — moved from CSS `content:` rule to React JSX in BoardPage. |

## Estimated 3-week plan if i18n becomes a priority

**Week 1 — Infra**
- Install i18next + react-i18next + i18next-browser-languagedetector
- Set up extraction (babel-plugin-i18next or i18next-parser)
- Wire CI to fail when a TSX file gains a hardcoded string
- Stub one locale (e.g. es-ES) with placeholder translations

**Week 2 — Extraction**
- Mechanical extraction of every JSX text node, aria-label, placeholder, error message
- Replace plural ternaries with `Trans` + count
- Replace hardcoded relative-time prose with `Intl.RelativeTimeFormat`
- Add currency + week-start to user preferences

**Week 3 — RTL + QA**
- Replace every `left:` / `right:` / `padding-left:` / `border-left:` with logical properties
- Add `dir="rtl"` toggle in account settings
- Translation review (real translators, not Google Translate)
- QA pass with each locale

---

## Why English-only is the right v1 call

- Users today: 1 (the builder)
- Marketing audience today: people the builder demos to (presumably English-speaking)
- Adding i18n on day 1 doubles the surface area of every change for zero current ROI
- Doing it well later is cheaper than doing it badly now and re-doing it

The right time to add i18n is the first time a real customer asks for it — not before.
